## A01:2025 — 访问控制失效（Broken Access Control）

位居第 1，100% 被测应用存在某种形式的访问控制失效。涵盖 40 个 CWE，SSRF 已并入本类别。

### 主要风险点
- 违反最小权限原则，默认允许而非默认拒绝
- 通过篡改 URL、参数、HTML 页面绕过访问控制检查
- 不安全的直接对象引用（IDOR），可查看/编辑他人账户
- POST/PUT/DELETE API 缺失访问控制
- 权限提升（未登录用户访问受保护页面，普通用户获得管理员权限）
- JWT 元数据篡改、Cookie 操纵
- CORS 配置错误导致 API 被非授权来源访问
- 强制浏览（猜测 URL）访问未授权页面

### 预防措施
- 非公开资源默认拒绝访问
- 集中实现访问控制机制，贯穿整个应用复用
- 访问控制应在服务端可信代码中实现
- 记录访问控制失败事件，对管理员发出告警
- 对 API 和控制器访问实施速率限制
- 注销后在服务端使会话标识符失效；JWT 令牌应短周期
- 单元测试和集成测试中包含功能性访问控制测试

## A02:2025 — 安全配置错误（Security Misconfiguration）

从 2021 年第 5 位上升至第 2 位，100% 被测应用存在某种配置错误，平均发生率 3.00%。涵盖 16 个 CWE，包括 XXE（CWE-611）。

### 主要风险点
- 应用栈任何层级缺乏安全加固，或云服务权限配置不当
- 启用了不必要的功能（端口、服务、页面、测试框架、特权）
- 默认账户和密码未修改
- 错误消息暴露堆栈跟踪等敏感信息
- 升级系统后最新安全功能未启用
- 过度兼容旧版本导致不安全配置
- 安全 HTTP 响应头缺失或配置不当

### 预防措施
- 建立可重复的安全加固流程，开发/QA/生产环境配置一致，仅凭证不同
- 最小化平台（移除不必要的功能、组件、文档、示例）
- 在补丁管理流程中审查并更新配置
- 应用分段架构实现有效隔离
- 向客户端发送安全指令（Security Headers）
- 使用自动化流程验证所有环境配置有效性
- 使用身份联合、短期凭证或平台提供的基于角色的访问机制，而非在代码中嵌入静态密钥

## A03:2025 — 软件供应链失效（Software Supply Chain Failures）

社区调查中 50% 受访者将其列为首要问题。从 2021 年的"已知漏洞组件"扩展为更广泛的供应链失效，平均发生率最高（5.72%），且拥有最高的平均漏洞利用和影响评分。涵盖 6 个 CWE。

### 主要风险点
- 未追踪所有组件版本（包括传递性依赖）
- 操作系统、框架、库、运行时等软件过时或不受支持
- 未定期扫描漏洞、未订阅安全公告
- 缺乏供应链变更管理（IDE、代码库、沙箱、镜像仓库等）
- 供应链未全面加固，缺乏最小权限
- CI/CD 管道安全性弱于所构建的系统
- 无职责分离（单人可将代码直接推送至生产）
- 使用来自不受信任来源的组件

### 预防措施
- 集中生成和管理整个软件的软件物料清单（SBOM）
- 追踪直接和传递性依赖
- 使用 OWASP Dependency Track、Dependency Check、retire.js 等工具持续清点组件
- 仅从官方可信来源通过安全链路获取组件，优先使用签名包
- 使用分阶段部署（金丝雀发布）限制供应商被攻破时的风险
- 对代码仓库、构建服务器、CI/CD、制品仓库全面加固并启用 MFA
- 构建服务器启用职责分离、签名构建、范围限定密钥

## A04:2025 — 加密失效（Cryptographic Failures）

从第 2 位降至第 4 位，涵盖 32 个 CWE，平均发生率 3.80%。常见弱点集中在弱随机数生成器和算法选择上。

### 主要风险点
- 使用过时或弱加密算法（MD5、SHA1、ECB 模式等）
- 默认加密密钥、弱密钥，或密钥管理/轮换缺失
- 加密密钥提交至代码仓库
- 未强制使用加密（HTTP 缺少安全头）
- 服务器证书未正确验证
- 初始化向量（IV）被重用或不安全
- 密码直接用作加密密钥而未通过 KDF 派生
- 使用不满足加密要求的随机性

### 预防措施
- 对数据分类并标记，确定需要加密的范围
- 使用硬件或云 HSM 存储最敏感的密钥
- 不存储不必要的敏感数据
- 使用 TLS >= 1.2，支持前向保密（FS），强制 HSTS
- 密码存储使用 Argon2、yescrypt、scrypt 或 PBKDF2-HMAC-SHA-512 等自适应哈希函数
- 始终使用认证加密（而非单纯加密）
- 现在即开始为后量子密码学（PQC）做准备，高风险系统需在 2030 年底前完成迁移

## A05:2025 — 注入（Injection）

从第 3 位降至第 5 位，100% 的应用被测试了某种形式的注入。涵盖 37 个 CWE，CVE 数量最多（62,445 个），包括 XSS（>30k CVE）和 SQL 注入（>14k CVE）。

### 主要风险点
- 用户提供的数据未经验证、过滤或净化
- 动态查询或无参数化调用直接传给解释器
- ORM 搜索参数中使用未净化数据
- 恶意数据直接拼接至动态查询或存储过程
- 常见注入类型：SQL、NoSQL、OS 命令、LDAP、EL/OGNL
- LLM 相关的提示词注入（见 OWASP LLM Top 10 LLM01）

### 预防措施
- 首选安全 API，完全避免使用解释器，或使用参数化接口/ORM
- 正向服务端输入验证（白名单）
- 对任何残留动态查询，使用特定于解释器的转义语法
- 在 CI/CD 管道中集成 SAST、DAST、IAST 工具
- 对所有参数、HTTP 头、URL、Cookie、JSON、SOAP、XML 输入进行自动化测试（包括模糊测试）

## A06:2025 — 不安全设计（Insecure Design）

从第 4 位降至第 6 位，涵盖 39 个 CWE，平均发生率 1.86%。2021 年首次引入，行业在威胁建模和安全设计方面已有改善。

### 主要风险点
- 缺乏业务风险分析，导致安全设计需求不明确
- 安全控制从未被设计（区别于实现缺陷）
- 缺乏威胁建模
- 业务逻辑缺陷（如未定义非预期状态变化）
- 需求阶段未收集安全和隐私保护需求
- 不安全的凭证存储（CWE-256）、不受限的文件上传（CWE-434）

### 预防措施
- 与 AppSec 专业人员建立安全开发生命周期（SDL）
- 建立安全设计模式库或"铺好路"的组件
- 对认证、访问控制、业务逻辑、关键流程进行威胁建模
- 在用户故事中集成安全语言和控制措施
- 在每个层级（前端到后端）集成合理性检查
- 为每个层级编写用例和误用用例，进行单元和集成测试

## A07:2025 — 认证失效（Authentication Failures）

维持第 7 位，名称略有调整（原为"身份认证和身份识别失效"）。涵盖 36 个 CWE，标准化认证框架的广泛使用已产生积极效果，但风险依然存在。

### 主要风险点
- 允许凭据填充攻击（使用已泄露的用户名/密码列表）
- 允许混合密码攻击（如 Winter2025 → Winter2026）
- 允许暴力破解或自动化脚本攻击
- 允许默认密码、弱密码或众所周知的密码
- 弱密码恢复流程（"安全问题"）
- 密码以明文、加密或弱哈希方式存储
- 缺失或无效的多因素认证（MFA）
- 会话 ID 暴露在 URL 中
- 登录成功后未更换会话 ID
- 注销或超时后未正确使会话/令牌失效

### 预防措施
- 尽可能实施并强制使用多因素认证（MFA）
- 鼓励并启用密码管理器
- 不部署任何默认凭据
- 实施弱密码检查（对照 Top 10,000 最差密码列表）
- 在创建账户和修改密码时验证已知泄露凭据（如 haveibeenpwned.com）
- 遵循 NIST 800-63b 的密码政策（长度/复杂度，不强制轮换）
- 注册、凭据恢复和 API 路径使用相同消息防止账户枚举
- 限制登录失败次数，记录失败事件并告警
- 使用服务端安全内置会话管理器，登录后生成新随机会话 ID

## A08:2025 — 软件和数据完整性失效（Software or Data Integrity Failures）

维持第 8 位，名称略有调整（"和"→"或"）。涵盖 14 个 CWE，专注于比供应链失效更低层级的完整性验证失败，特别是不安全的反序列化。

### 主要风险点
- 依赖不受信任来源的插件、库或 CDN 资源
- 不安全的 CI/CD 管道（未验证代码/制品完整性）
- 自动更新功能未进行充分完整性验证
- 不安全的反序列化（Java 对象、JSON 等可被攻击者修改）
- 从不受信来源下载包（无签名验证）

### 预防措施
- 使用数字签名验证软件或数据来源及完整性
- 确保 npm/Maven 等依赖仅使用可信仓库，高风险场景考虑自建私有仓库
- 建立代码和配置变更审查流程
- CI/CD 管道实施适当的隔离、配置和访问控制，确保构建部署流程完整性
- 对来自不受信客户端的序列化数据，在使用前进行完整性检查或数字签名验证

## A09:2025 — 安全日志与告警失效（Security Logging & Alerting Failures）

维持第 9 位，名称调整（原"监控"→"告警"）以强调告警功能的重要性。第三次通过社区调查投票入选，难以在测试中发现（仅 723 个 CVE），但对事件检测和响应的影响极大。

### 主要风险点
- 可审计事件（登录、失败登录、高价值交易）未记录或记录不一致
- 警告和错误日志消息不足或不清晰
- 日志完整性未受保护（可被篡改）
- 应用/API 日志未监控异常活动
- 日志仅本地存储，无备份
- 告警阈值和响应升级流程缺失或无效
- 渗透测试和 DAST 扫描未触发告警
- 日志中泄露敏感信息（PII、PHI）
- 日志数据未正确编码导致注入风险
- 假阳性告警过多导致 SOC 团队负担过重

### 预防措施
- 记录所有登录、访问控制、服务端输入验证失败事件，保留足够时间以支持事后取证分析
- 确保包含安全控制的每个应用部分都有日志记录（成功和失败）
- 以日志管理系统可消费的格式生成日志，并正确编码
- 为日志提供防篡改审计跟踪（如仅追加的数据库表）
- 在应用中加入"蜜标（honeytokens）"作为攻击者陷阱
- 建立有效的监控告警用例和响应手册（SOC playbook）
- 建立并采用事件响应和恢复计划（如 NIST 800-61r2）

## A10:2025 — 异常条件处理不当（Mishandling of Exceptional Conditions）

2025 年新增类别，涵盖 24 个 CWE，专注于不当错误处理、逻辑错误、"失败打开（failing open）"等由异常条件引起的问题。

### 主要风险点
- 应用未防止异常情况发生（缺乏输入验证、资源配额、速率限制）
- 应用未识别正在发生的异常情况
- 应用对异常情况响应不当或无响应
- 错误消息包含敏感系统信息（供攻击者侦察）
- 异常被捕获但资源未释放（导致资源耗尽/DoS）
- 多步骤事务在出错后未回滚（状态损坏）
- "失败打开"：异常发生时系统未拒绝而是允许操作继续

### 预防措施
- 在错误发生的具体位置直接捕获并处理每个可能的系统错误
- 处理包含：向用户抛出可理解的错误、记录事件日志、必要时发出告警
- 设置全局异常处理器作为兜底
- 事务执行中途出错必须完整回滚（"失败关闭"原则）
- 在任何可能的地方添加速率限制、资源配额、节流等防护
- 集中化错误处理、日志记录、监控和告警（同一应用使用一种机制）
- 严格的输入验证（白名单 + 危险字符净化或转义）
- 执行压力测试、性能测试和渗透测试

## 2025 版主要变化

- **A03（软件供应链失效）** 扩展自 2021 年的"已知漏洞组件"，覆盖更广泛的供应链攻击面
- **A10（异常条件处理不当）** 为全新类别，源自不当错误处理专项拆分
- **SSRF** 从独立类别并入 A01（访问控制失效）
- 数据来源：280 万+ 应用，589 个分析 CWE，248 个入选 Top 10
