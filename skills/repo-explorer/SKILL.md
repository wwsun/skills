# Repo Explorer

帮助用户在 10-15 分钟内建立对陌生代码库的系统性认知，生成一份导航地图式的项目报告。

**输出方式**：默认把报告输出到对话中。只有当用户明确说"保存到文件"或"写到 PROJECT_NOTES.md"时才写文件——不要自作主张创建文件，大多数时候用户只需要在对话里看到结果。

## 工作流程

按以下 5 个阶段探索，探索时**广度优先**：先勾勒轮廓，再深入关键细节，避免过早陷入某个文件。

---

### 阶段 1：项目身份识别（快速，约 2 分钟）

并行读取这些文件（有则读，没有跳过）：

**配置文件（判断技术栈）**
- `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml` / `pom.xml`
- `.nvmrc` / `.tool-versions` / `Makefile` / `Dockerfile`

**项目描述**
- `README.md`（只读前 100 行，足够了解项目定位）
- `ARCHITECTURE.md` / `docs/` 目录（如果存在）

从这些文件中提取：
- 这个项目**是什么**（一句话定位）
- 用什么语言和框架
- 谁在用它（工具/库/应用？面向开发者还是终端用户？）

**Monorepo 检测**：如果发现 `pnpm-workspace.yaml`、`lerna.json`、`nx.json`、`turbo.json`，或 `packages/`/`apps/` 目录下有多个子包，这是 monorepo。需要：
1. 列出所有子包名称和简单用途
2. 识别哪个包是用户最可能关心的（主包/核心包）
3. 重点探索核心包，其他包点到为止

---

### 阶段 2：目录结构扫描（快速，约 2 分钟）

列出顶层目录，然后对每个有意义的目录再列一层。跳过：`node_modules`、`.git`、`dist`、`build`、`coverage`、`__pycache__`、`.cache`。

识别目录角色（参考常见模式）：

| 目录名 | 通常代表 |
|--------|----------|
| `src/` `lib/` `pkg/` | 核心源码 |
| `app/` `pages/` `routes/` | 应用层/路由 |
| `api/` `server/` `handlers/` | 服务端逻辑 |
| `components/` `ui/` | UI 组件 |
| `utils/` `helpers/` `common/` | 工具函数 |
| `hooks/` | React Hooks |
| `store/` `state/` `context/` | 状态管理 |
| `tests/` `__tests__/` `spec/` | 测试 |
| `scripts/` `bin/` | 构建/工具脚本 |
| `docs/` | 文档 |
| `examples/` | 示例代码 |
| `cli/` `cmd/` | 命令行入口 |
| `packages/` `apps/` | Monorepo 子包 |

---

### 阶段 3：入口点定位（重点，约 3 分钟）

找到代码从哪里"开始执行"。不同类型项目的入口点不同：

**JS/TS 项目**
- 看 `package.json` 的 `main`、`exports`、`bin` 字段
- 框架约定：Next.js→`app/`或`pages/`，Express→`server.js`/`app.js`，CLI→`bin/`
- 如果是库：找 `index.ts` / `index.js`

**Python 项目**
- `__main__.py`、`main.py`、`setup.py` 的 `entry_points`
- `pyproject.toml` 的 `[tool.poetry.scripts]`

**Go 项目**
- `main.go` 或 `cmd/*/main.go`

**通用**
- `Makefile` 的 `run` / `start` target
- `Dockerfile` 的 `CMD` / `ENTRYPOINT`

读入口文件，弄清楚它初始化了什么、串联了哪些模块。

---

### 阶段 4：核心模块和数据流（深度，约 5 分钟）

**找到核心业务逻辑**

从入口点开始，沿着调用链找到"实质性工作"在哪里发生。读 3-5 个最关键的文件（每个读核心部分，不需要逐行读完）。

重点理解：
1. **请求/数据从哪里进来** → 经过哪些处理 → 从哪里出去
2. **模块之间如何通信**（直接调用？事件？消息队列？）
3. **是否存在状态机或生命周期**（如请求处理阶段、组件生命周期、任务状态流转）

**核心抽象识别**

找出项目中最重要的 3-5 个概念/类型，理解它们是什么、职责边界在哪里、彼此关系如何。典型的核心抽象包括：
- 核心领域对象（如 `Agent`、`Session`、`Skill`、`Provider`）
- 核心接口/协议（如 `ModelProvider`、`Transport`、`Installer`）
- 核心数据结构（如 lock 文件格式、配置 schema、消息格式）

读对应的 `types.ts`、`interfaces/`、核心 class 定义，提取关键字段和方法签名（不需要实现细节）。

**数据流图格式**：用多级缩进的 ASCII 图表达，让人一眼看清层次关系。例：

```
用户输入源地址
  → source-parser.ts: parseSource()     # 判断 github/gitlab/local/well-known
  → [远程] git.ts: cloneRepo()          # clone 到临时目录
  → skills.ts: discoverSkills()         # 递归找 SKILL.md，gray-matter 解析
  → 交互选择（@clack/prompts）
  → installer.ts: installSkillForAgent() # symlink 或 copy
  → skill-lock.ts: updateLock()         # 更新锁文件
```

比起单行箭头（`A → B → C`），这种格式能展示每一步"做什么"，更有价值。

**依赖分析**

列出关键外部依赖，说明**为什么用它**，不只是列名字。区分运行时依赖 vs 开发依赖 vs 可选依赖。

### 阶段 5：生成导航报告

将探索结果组织成以下格式，输出到对话中。

---

## 报告结构

```markdown
# [项目名] 架构速览

## 一句话定位
[这个项目是什么，解决什么问题，面向谁]

## 技术栈
- 语言：
- 框架：
- 构建工具：
- 测试框架：
- 主要运行环境：

## 目录结构
[带注释的目录树，只保留有意义的部分，注释说明"为什么"而不只是"是什么"]

## 入口点
- 主入口：[文件路径]（做了什么）
- CLI 入口：[如果有]
- API 入口：[如果有]

## 核心数据流
[用多级缩进 ASCII 图展示主要请求/数据的流转路径，每步标注文件名和函数名]

## 整体架构
[说明模块间的调用关系和分层结构。如果存在状态机或生命周期（如请求处理阶段、任务状态流转），用简单图表展示状态和转换条件]

例（分层架构）：
```
CLI 层        cli.ts → 命令路由
Service 层    add.ts / remove.ts → 业务编排
Core 层       installer.ts / skills.ts → 核心逻辑
Adapter 层    git.ts / providers/ → 外部系统适配
```

例（状态机）：
```
[IDLE] → parseSource() → [FETCHING] → cloneRepo() → [READY]
                                              ↓ 失败
                                          [ERROR] → 清理临时目录
```

## 核心抽象
[列出 3-5 个最重要的概念/类型，说明职责和关键字段/方法]

例：
- **`Agent`**（`src/agents.ts`）— 表示一个 AI 编码工具，定义了它的技能目录路径和安装检测逻辑
  - 关键字段：`id`、`name`、`skillsDir`、`isUniversal`
- **`Skill`**（`src/types.ts`）— 一个可安装的技能单元，来自 SKILL.md 解析
  - 关键字段：`name`、`description`、`path`、`frontmatter`
- **`LockEntry`**（`src/skill-lock.ts`）— lock 文件中的一条记录，追踪已安装技能的来源和版本
  - 关键字段：`source`、`folderHash`、`installedAt`

## 关键文件速查
| 想了解什么 | 去哪里找 |
|------------|----------|
| [基于项目实际内容填写，不要套模板] | [具体文件路径或目录] |

## 核心依赖
| 库 | 用途 |
|----|------|
| xxx | yyy（为什么选这个而不是其他方案） |

## 值得关注的设计决策
[仅在发现有趣的架构选择时才写这一节，比如：为什么不用某主流库？某个不寻常的分层方式？某个隐藏的扩展点？]
[如果没有特别值得一提的，跳过这节]
```

## 报告生成原则

**要做的：**
- 目录树注释要说明"为什么"而不只是"是什么"（例：`# 统一处理错误，避免每个路由重复写 try-catch`）
- 数据流描述要具体到文件和函数级别
- 关键文件速查表要基于这个项目实际有什么，不要套通用模板
- "值得关注的设计决策"只写真的有趣的发现，没有就不写

**不要做的：**
- 不要试图读完所有文件
- 不要在报告里复制粘贴代码片段（路径 + 说明就够了）
- 不要对用户没问的东西发表评判（比如"这段代码写得不好"）
- 不要生成虚假的目录结构（只写实际存在的）
- 不要未经用户要求就保存文件

---

## 后续对话模式

生成报告后，主动提示用户可以继续深挖：

> "报告已生成。你可以继续问我：
> - '认证是怎么实现的？'
> - '找找哪里处理了 webhook 事件'
> - '解释一下 X 模块和 Y 模块的关系'
> - 或者直接说你想了解的功能点"

在后续对话中，基于已有的探索结果快速定位，不需要重新扫描整个项目。
